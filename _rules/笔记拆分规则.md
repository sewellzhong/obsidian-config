# 笔记拆分规则

本文是知识库笔记文件的拆分规则。

## 1. 分层定义（L0-L99）

- L0：索引与入口（目录、清单、导航、MOC）
  - 判定问题：是否仅用于导航与聚合？
- L1：原子知识（单一概念/结论/原则）+ 最小理解示例
  - 判定问题：是否可独立复用的一条最小知识？
- L2：结构模型（多个 L1 之间的关系、对比、框架、系统图）
  - 判定问题：是否在描述多条 L1 之间的关系结构？
- L3：过程与实践（步骤、案例、经验、项目流、教程示例）
  - 判定问题：是否以时间顺序或实践过程为主线？
- L99：复盘（日、周、月）
  - 判定问题：是否以复盘作为内容输入来源？

## 2. 拆分准则

- 单一意图：一条笔记只解决一个核心问题
- 原子化：出现多个并列主题时必须拆分
- 抽象隐含概念：从案例或过程提取可复用的 L1
- 结构化：将多个 L1 组织为 L2（关系、对比、层级、因果）
- 双向链接：新笔记必须至少与 1 条相关笔记互链
  - 例外：新领域首条笔记可仅链接 MOC 或允许 0 链接
- MOC 更新：新增 L2/L3 时给出 MOC 更新建议
- 互链规范：链接需指向具体笔记文件，避免仅链接到同文件段落

## 3. 拆分停止条件

- 内容能在一页内自洽
- 只包含一个核心意图
- 再拆分会破坏上下文或降低可读性
- 建议阈值：正文不超过 600 字或 8 段（超过则优先检查可拆分点）

最小理解示例格式：
- 定义/结论
- 正例或反例至少 1 个

## 4. 命名规范

- 命名格式：`LXX【领域｜对象｜作用类型】一句话标题`
- 对象类型建议：概念、模型、方法、工具、流程、案例、问题
- 对象要求：使用名词短语，避免动词开头；示例：`概念/递归`、`工具/Git`、`方法/番茄工作法`
- 标题要求：一句话可读，避免模糊词

## 5. 标签规范（三维必须齐全）

- 知识领域维度：`#领域/...`（至少二级，如 `#领域/计算机科学/软件开发`）
  - 例外：新领域建立期可暂用一级领域标签，后续补全二级
- 作用类别维度：`#作用/...`
- 层级维度：`#层级/L0` `#层级/L1` `#层级/L2` `#层级/L3` `#层级/L99`
- 标签来源：`标签索引系统.md`
- 新增标签：需要更新本规则并同步到文件 `标签索引系统.md`
- ⚡标签：笔记的补充标签，用于快速索引笔记
  - 可选：笔记不一定需要该类标签

## 6. 元数据规范

- 格式：YAML frontmatter，数组形式
- 包含字段：title/aliases/tags
  - title：标题。字段必须存在，内容为一句话标题
  - aliases：别名。字段必须存在，内容可选（可包含简称）
  - tags：标签。必须存在且包含三维标签

## 7. 质量检查（触发时机）

- 关系推断、冗余检测、粒度审查：新增 L2/L3 时执行
- 知识熵评估、概念演化追踪：同领域新增 3 条 L1 后执行
- MOC 更新触发：新增 L2/L3 必须更新；新增 L1 若已满 5 条则更新
- MOC 目标：优先指向对应领域 MOC；若不存在则新建并按命名规范命名

## 8. 输出结构（处理原始输入时）

- 笔记整理：分析现有知识库笔记内容，进行笔记整理
  - 分析笔记：按命名规范命名的笔记
- 输出笔记文件：按命名规范新建文件
  - 文件位置：原笔记位置下新建目录
  - 笔记清单：文件名
- 给出 MOC 更新建议
- 只输出结果，禁止解释
- 必须包含以下板块（无内容也需保留标题）：
  - 📍 MOC更新建议
  - 🧬 思维模式推断
  - 📈 概念成熟度
  - 💰 知识ROI评估
  - 🏗 知识库重构建议
  - 🧠 个体认知模型
  - 🧪 探索任务
  - 🪜 能力瓶颈定位
  - 🌌 抽象推进建议
- 最简模式（无需认知评估时）：
  - 仅输出“笔记清单 + MOC 更新建议”
  - 其余板块保留标题即可
- 触发条件：
  - 完整模式：输入为原始资料或未结构化内容
  - 最简模式：输入已结构化或用户明确要求简化输出

## 9. 规则更新流程

- 修改规则需同步更新相关索引文件（如 `标签索引系统.md`）
